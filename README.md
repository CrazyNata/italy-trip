# Отпуск в Италии

Планировщик семейной поездки в Италию (25 сентября — 12 октября 2026): маршрут по дням, жильё, сроки бесплатной отмены, места, бюджет и заметки.

Проект переезжает со статического legacy-приложения на React. Сейчас в корне находится production-фундамент на React, TypeScript, Vite и Tailwind CSS с Supabase Auth, ролями владельца/читателя, настройками профиля и синхронизацией данных поездки; перенос интерфейса разделов выполняется отдельными этапами.

## Требования

- Node.js 22 или новее
- npm 10 или новее

## Локальный запуск

```bash
npm install
npm run dev
```

Production-сборка:

```bash
npm run build
npm run preview
```

Vite публикует приложение с базовым путём `/italy-trip/`, соответствующим адресу проекта на GitHub Pages.

## Конфигурация

Публичные URL проекта Supabase и publishable key имеют безопасные значения по умолчанию из предыдущего клиента. Их можно переопределить локально:

```dotenv
VITE_SUPABASE_URL=https://example.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_example
```

Переменные `VITE_*` попадают в клиентскую сборку и не должны содержать секреты. После входа токен Mapbox загружается из строки `mapbox_token` таблицы `app_config`; секреты в Git и Vite-переменные добавлять нельзя.

После применения миграции `supabase/migrations/20260711120000_normalized_collaboration.sql` React загружает доступные пользователю нормализованные поездки. Кэш хранится отдельно для каждой поездки в `localStorage.italy_trip:<user-id>:<trip-id>`, а право редактирования определяется ролью в `trip_members`. Если нормализованных поездок ещё нет, приложение читает прежнюю строку `trip_state.main` и `localStorage.italy_trip` как fallback, не изменяя их.

### Переход на нормализованные данные

1. Просмотрите и примените SQL-миграцию штатным процессом Supabase до публикации нового frontend. Репозиторий не применяет её автоматически.
2. Войдите под пользователем, чей email первым совпадает с `admins` при сортировке по email и UUID.
3. Один раз выполните `select public.import_legacy_main_trip();` в авторизованном контексте этого пользователя. Повторный вызов безопасно возвращает ту же поездку благодаря маркеру `legacy_trip_imports.source = 'trip_state:main'`.
4. Проверьте поездку в интерфейсе и только затем публикуйте frontend.

`trip_state.main` не обновляется и не удаляется миграцией или новым клиентом, поэтому rollback возвращает предыдущий frontend к исходному payload. Приглашения содержат одноразовый токен только в созданной владельцем ссылке; в базе хранится его SHA-256. Фото лежат в закрытом bucket `trip-photos` по пути `<trip-uuid>/<object>` и доступны по политике участника поездки.

## Структура

- `src/app/` — корень React-приложения
- `src/auth/` — авторизация, проверка роли и интерфейс аккаунта
- `src/appearance/` — темы и тёмный режим
- `src/components/` — общие компоненты и оболочка
- `src/trip/` — нормализованный repository, агрегирование данных, выбор поездки, приглашения, кэш и Realtime
- `src/features/` — будущие функциональные разделы поездки
- `src/lib/supabase/` — конфигурация клиента Supabase
- `src/styles/` — глобальные стили, шрифты и дизайн-переменные
- `src/types/` — глобальные TypeScript-типы
- `public/images/` — сохранённые фотографии; Vite копирует их в `dist/images/` без изменения URL
- `supabase/migrations/` — проверяемые SQL-миграции; они не применяются автоматически
- `legacy/` — исходные runtime-файлы приложения до миграции

## Legacy и откат

`legacy/` сохраняет старые `index.html`, `auth.js`, `sync.js`, `support.js` и `image-slot.js` как справочный снимок. Он не участвует в Vite-сборке и не публикуется как отдельное приложение. Полный проверенный откат выполняется через Git bundle и историю репозитория; удалённый legacy-тест намеренно не сохранён.

## Деплой

Push в `main` запускает `.github/workflows/deploy.yml`: workflow устанавливает Node.js 22, выполняет `npm ci`, собирает приложение и публикует только `dist/` в GitHub Pages. Vite добавляет content hash к production-ресурсам, поэтому прежняя подстановка `__BUILD__` больше не нужна.
